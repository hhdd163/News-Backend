本项目已完成以下模块的缓存优化：

- ✅ 新闻列表缓存

## 🔑 缓存键命名规则

### News 模块
```
news:categories                          # 新闻分类
news:list:{category_id}:{page}:{size}    # 新闻列表
news:detail:{news_id}                    # 新闻详情
news:related:{news_id}                   # 相关推荐
news:views:{news_id}                     # 浏览量计数器
```

### User 模块
```
user:info:{user_id}                      # 用户信息
```

### Favorite 模块
```
favorite:check:{user_id}:{news_id}       # 收藏状态（"1"或"0"）
```

## ⏱️ 缓存过期时间配置

| 缓存类型 | 过期时间 | 说明 |
|---------|---------|------|
| 新闻分类 | 7200秒（2小时） | 数据稳定，缓存时间长 |
| 新闻列表 | 1800秒（30分钟） | 中等稳定性 |
| 新闻详情 | 1800秒（30分钟） | 排除 views 字段 |
| 相关推荐 | 900秒（15分钟） | 推荐结果相对稳定 |
| 用户信息 | 3600秒（1小时） | 用户信息不常修改 |
| 收藏状态 | 300秒（5分钟） | 用户可能频繁操作 |
| 浏览量计数 | 86400秒（24小时） | 防止内存泄漏 |

## 💡 缓存工作流程

### 1. 新闻详情缓存流程
```
用户请求新闻详情
  ↓
检查缓存 news:detail:{id}
  ↓
缓存命中？
  ├─ 是 → 返回缓存数据 + 实时查询 views
  └─ 否 → 查询数据库 → 写入缓存（排除views）→ 返回
```

### 2. 收藏状态缓存流程
```
用户请求检查收藏状态
  ↓
检查缓存 favorite:check:{uid}:{nid}
  ↓
缓存命中？
  ├─ 是 → 直接返回（"1"或"0"）
  └─ 否 → 查询数据库 → 写入缓存 → 返回
```

### 3. 用户信息缓存流程
```
用户请求（需要认证）
  ↓
验证 Token 获取 user_id
  ↓
检查缓存 user:info:{uid}
  ↓
缓存命中？
  ├─ 是 → 使用缓存数据填充 user 对象
  └─ 否 → 使用数据库数据 → 写入缓存
```

   - 提升详情页加载速度

### 监控建议

使用 Redis 命令监控缓存命中率：
```bash
# 查看缓存统计
redis-cli info stats

# 关注指标
keyspace_hits    # 缓存命中次数
keyspace_misses  # 缓存未命中次数
hit_rate = hits / (hits + misses)  # 命中率
```

## ⚠️ 注意事项

### 1. 缓存一致性
- 数据更新时必须清除相关缓存
- 避免缓存与数据库数据不一致

### 2. 缓存雪崩预防
- 不同数据设置不同过期时间
- 已实现：分类(7200s)、列表(1800s)、详情(1800s)

### 3. 缓存穿透保护
- 对于不存在的 key，缓存空值
- 设置较短过期时间（如60秒）

### 4. 内存管理
- 定期监控 Redis 内存使用
- 浏览量计数器设置24小时过期

## 🔧 故障排查

### 问题1：Redis 连接失败
```
检查：
1. Redis 服务是否启动
2. 配置是否正确（config/cache_config.py）
3. 端口是否被占用
```

### 问题2：缓存不生效
```
检查：
1. 缓存键是否正确
2. 过期时间是否设置
3. 查看应用日志是否有错误信息
```

### 问题3：缓存数据不一致
```
解决：
1. 手动清除缓存
redis-cli> del user:info:1
2. 检查缓存失效逻辑是否正确
```
